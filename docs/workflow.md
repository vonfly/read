# 功能开发工作流

## 核心原则

先定义清楚"做什么"，再让 AI 写代码。每个功能模块都从一个 spec 文件开始，
而不是直接开口让 AI 实现。

---

## 完整流程

```
你写 Feature Spec（5-10 分钟）
        ↓
Claude Code Plan 模式：生成 PLAN.md（审查后再执行）
        ↓
Claude Code 按 plan 执行（串行逐文件 或 并行多 Agent）
        ↓
每个文件编译通过后继续下一个
        ↓
对照 Spec 验收标准逐条人工验收
        ↓
通过后归档 spec + plan 文件
        ↓
推送到 GitHub
```

---

## Step 1：写 Feature Spec

在 `docs/features/` 目录下为每个功能新建一个 spec 文件。

**文件命名：** `[模块名]-spec.md`，例如 `booklist-spec.md`、`reader-spec.md`

**模板：**

```markdown
# [功能名称] Spec

## 要解决的问题
描述用户痛点或需求背景。

## 设计规格
> 以下数据来自 Pencil 设计稿 `[文件名].pen` → `[页面名称]` 页面

### 页面整体
| 属性 | 值 |
|------|-----|
| 宽度 | xxx dp |
| 高度 | xxx dp |
| 背景 | `$--变量名` |
| 布局 | 垂直/水平 |

### 组件名称（如 Header）
| 属性 | 值 |
|------|-----|
| 高度 | xx dp |
| 内边距 | xx dp |
| 圆角 | xx dp |
| 背景 | `$--变量名` |

**子元素（如标题）**
| 属性 | 值 |
|------|-----|
| 内容 | "xxx" |
| 字体 | Inter |
| 字号 | xx sp |
| 字重 | xxx |
| 颜色 | `$--变量名` |

（按此格式列出所有组件和子元素的设计规格）

## 验收标准

### UI 验收（视觉还原）
- [ ] 页面背景色使用 `$--变量名` 变量
- [ ] 所有卡片圆角 xx dp
- [ ] 文本字体、字号、字重符合设计规格
- [ ] 间距、内边距符合设计规格
- [ ] 图标尺寸、颜色符合设计规格

### 功能验收
- [ ] 功能行为可验证的条目（如：点击书籍跳转阅读器并传入正确 bookId）
- [ ] 边缘情况的处理结果（如：书架为空时显示占位图和导入按钮）
- [ ] 错误状态的呈现（如：网络失败显示错误提示和重试按钮）

### 数据验收
- [ ] 数据来源正确（Room / API / Mock）
- [ ] 数据为空时正常显示

## 实现对照表（Agent 必须严格遵循）

> ⚠️ 以下数值必须精确实现，禁止使用"习惯值"或主题变量默认值

### 组件名称（如 BookmarkCard）

| 属性 | 设计值 | Compose 实现 | 状态 |
|------|--------|--------------|------|
| 卡片内边距 | 12dp | `.padding(12.dp)` | ⬜ |
| 封面尺寸 | 52×72dp | `.size(width = 52.dp, height = 72.dp)` | ⬜ |
| 封面圆角 | 4dp | `RoundedCornerShape(4.dp)` | ⬜ |
| 元信息颜色 | $--foreground-tertiary | `ForegroundTertiary` | ⬜ |
| 预览文本颜色 | $--foreground-secondary | `ForegroundSecondary` | ⬜ |
| 时间颜色 | $--foreground-tertiary | `ForegroundTertiary` | ⬜ |

（按此格式列出所有需要精确实现的数值）
```

**填写原则：**
- **设计规格必须详细**：通过 Pencil MCP 读取设计稿后，将所有尺寸、颜色、字体、间距等规格写入 spec
- **使用表格格式**：组件属性用表格呈现，清晰易读
- **UI 结构树**：用树形结构展示组件层级关系，便于理解页面结构
- **验收标准分三类**：UI 验收（视觉还原）、功能验收（行为正确）、数据验收（数据来源）
- "不在本期范围内"最重要，AI 倾向于过度实现，明确排除能节省大量返工
- 技术约束必须填"数据来源"，其余路径、分层、状态管理由 CLAUDE.md 和 agent 保证

## Step 2：Plan 模式生成计划文件

打开 Claude Code，**按 Shift+Tab 切换到 Plan 模式**，输入：

```
读取 docs/features/[功能名]-spec.md，
同时通过 Pencil MCP 读取设计稿中"[页面名称]"页面（无设计稿则删除此行），
结合项目现有架构，把这个功能拆分成具体的实现任务。
将完整计划写入 docs/features/[功能名]-plan.md，不要立即执行任何代码。
等我阅读完成 docs/features/[功能名]-plan.md 后，确认没有问题我会退出 Plan 模式，再通知你执行

计划文件格式要求：
- 按依赖关系分批，同批内的任务互不依赖可并行，下一批依赖上一批完成
- 使用表格格式追踪进度，便于中断后继续
- 状态符号：⬜ 待开始 | 🔄 进行中 | ✅ 已完成 | ❌ 编译失败

**计划文件模板：**

```markdown
# [功能名称] - 实现计划

## 状态概览
- 创建时间：YYYY-MM-DD
- 进度：0/N (0%)

---

## 第一批：领域层（可并行）

| 状态 | 文件路径 | 职责 |
|------|----------|------|
| ⬜ | domain/model/Book.kt | 书籍数据模型 |
| ⬜ | domain/repository/BookRepository.kt | Repository 接口 |

---

## 第二批：数据层（依赖第一批）

| 状态 | 文件路径 | 职责 |
|------|----------|------|
| ⬜ | data/local/entity/BookEntity.kt | Room Entity |
| ⬜ | data/repository/BookRepositoryImpl.kt | Repository 实现 |

---

## 第三批：UI 层（依赖第二批）

| 状态 | 文件路径 | 职责 |
|------|----------|------|
| ⬜ | ui/screen/booklist/BookListViewModel.kt | ViewModel |
| ⬜ | ui/screen/booklist/BookListScreen.kt | Screen + Content |
```
```

> Pencil MCP 需要提前打开 Pencil 并加载设计稿文件，Claude Code 才能读取。

**为什么先生成 PLAN.md 再执行？**
- 任务中断或 session 超时后，可以直接读取计划文件继续，不用重新规划
- 在代码动工之前发现架构误解，比写完一半再返工代价小得多
- 计划文件和 spec 一起归档，记录每个功能的决策过程

**审查 `[功能名]-plan.md` 时逐条确认：**
- [ ] 文件顺序是否合理（通常：domain → data → ui，从底层到顶层）
- [ ] 每个文件的职责描述是否和 spec 的验收标准对得上
- [ ] 有没有遗漏的文件（如忘记更新 `AppNavHost.kt`）
- [ ] 有没有多余的文件（AI 有时会生成不必要的抽象层）
- [ ] 涉及新增依赖时，库名是否已在 `libs.versions.toml` 中存在

确认无误后，再按 Shift+Tab 退出 Plan 模式，开始执行。

## Step 3：执行计划

### 默认方式：串行执行（推荐新手）

```
按照 docs/features/[功能名]-plan.md 的计划
```

Claude Code 会依次实现每个文件，每次自动运行编译命令验证，编译通过后自动推进，
编译失败则自行修复，直到通过再继续。全程无需人工干预。

**状态更新规则：**
- 开始文件前：将状态从 ⬜ 改为 🔄
- 编译通过后：将状态从 🔄 改为 ✅
- 编译失败时：将状态从 🔄 改为 ❌，修复后改为 ✅
- 每批完成后：更新顶部进度（如 `3/8 (37%)`）

**禁止跳步：** 编译失败时不得继续下一个文件，必须修复当前文件后再推进。

### Agent 执行提示词规范

**启动 Agent 时，必须包含以下约束：**

```
按照 docs/features/[功能名]-plan.md 的计划实现。

**UI 实现要求（必须严格遵循 spec 文档）：**
1. 禁止使用"习惯值"（如 16dp、48dp），必须使用 spec 中的精确值
2. 禁止使用主题变量替代设计值（如 Spacing.Lg 替代 12dp）
3. 颜色必须严格对照 spec：
   - $--foreground-secondary → ForegroundSecondary
   - $--foreground-tertiary → ForegroundTertiary
4. 每个组件实现后，对照 spec 中的设计规格逐项验证

**常见错误预警：**
- 内边距：设计稿常用 12dp，不要默认用 16dp
- 封面尺寸：严格使用 spec 中的宽×高，不要用"习惯尺寸"
- 颜色：secondary 和 tertiary 不要混淆
```

## Step 4：验收与归档

所有文件编译通过后，打开模拟器逐条验证 spec 中的验收标准：

**验收清单（每条必须人工确认）：**

### UI 视觉验收
- [ ] 页面整体尺寸、背景色符合设计规格
- [ ] 所有组件尺寸、间距、内边距符合设计规格
- [ ] 文本字体、字号、字重、颜色符合设计规格
- [ ] 图标尺寸、颜色符合设计规格
- [ ] 卡片圆角、阴影效果符合设计规格
- [ ] 颜色变量使用正确（`$--primary`、`$--background` 等）
- [ ] 深色/浅色主题切换后颜色正确变化（如适用）

### 功能验收
- [ ] spec 中每条功能验收标准是否全部通过
- [ ] 点击交互响应正确（跳转、Toast 等）
- [ ] 状态切换正确（如夜间模式开关）

### 数据验收
- [ ] spec 中每条数据验收标准是否全部通过
- [ ] 空状态是否正确显示（空书架、无网络、加载失败）
- [ ] 数据格式正确（如数字显示、单位转换）

### 边缘情况
- [ ] 快速操作不会导致崩溃或异常
- [ ] 重复点击不会触发多次操作
- [ ] 横竖屏切换后布局正常
- [ ] 不同尺寸模拟器上布局正常（手机 / 平板）

### UI 常见错误检查清单（重点！）

> ⚠️ 以下是最容易出错的"习惯值"问题，必须逐项检查：

**尺寸与间距**
- [ ] 卡片内边距：设计稿是 12dp 还是 16dp？
- [ ] 封面尺寸：严格对照 spec 中的 宽×高（如 52×72dp，不是 48×64dp）
- [ ] 组件间距：设计稿是 12dp 还是 Spacing.Md（16dp）？
- [ ] 列表间距：设计稿是 12dp 还是 8dp？

**颜色映射**
- [ ] 元信息（作者·章节）：`ForegroundTertiary` 不是 `ForegroundSecondary`
- [ ] 预览文本：`ForegroundSecondary` 不是 `ForegroundTertiary`
- [ ] 时间戳：`ForegroundTertiary`
- [ ] 主色调文字：`Foreground`

**字体规格**
- [ ] 标题字号：15sp 还是 16sp？
- [ ] 元信息字号：12sp 还是 13sp？
- [ ] 预览文本字号：13sp，行高 1.3

**布局结构**
- [ ] 卡片高度：设计稿是固定值（如 96dp）还是 wrap_content？
- [ ] Header 高度：44dp（自定义 Row）还是使用 TopAppBar？

> AI 无法替代这一步。即使 AI 声称"已验证通过"，仍需人工在模拟器上亲自操作确认。

**全部通过后归档：**

```
将以下文件移至 docs/features/done/ 目录：
- docs/features/[功能名]-spec.md
- docs/features/[功能名]-plan.md
```

## 任务中断后如何继续

如果 session 超时或被打断，下次继续时：

```
读取 docs/features/[功能名]-plan.md，从第一个 ⬜ 或 🔄 或 ❌ 的任务继续
```

AI 会自动：
1. 读取 plan 文件，识别已完成（✅）和未完成（⬜/🔄/❌）的任务
2. 从第一个未完成任务开始继续实现
3. 每完成一个文件更新 plan 文件状态

**不需要手动列出已完成的文件**，plan 文件的状态列就是进度追踪器。

```
001-项目初始化          → docs/setup.md
002-书架列表            → 书籍展示、阅读进度
003-本地导入            → EPUB/TXT 文件导入、解析
004-阅读器核心          → 翻页、进度记录

008-账号与云同步（可选）→ 多端同步
```

每个模块完成并验收后再开始下一个。

## 常见问题

**Q：AI 生成的代码和 spec 不符怎么办？**
把 spec 中对应的验收标准贴给 AI，说明哪里不符合，让它修正。

**Q：编译报错看不懂怎么办？**
直接把完整报错信息贴给 AI：`这是编译错误，请修复：[错误信息]`

**Q：一个功能太大，一个 spec 写不下怎么办？**
按用户可感知的子功能拆分，每个子功能一个 spec，分多轮开发。

**Q：需求中途变化怎么办？**
先更新 spec 文件，再告诉 AI spec 已更新，同时更新 plan.md 中受影响的任务项。
不要在不更新 spec 的情况下直接让 AI 改代码，会导致 spec、plan 和代码三者不一致。
